24 /*
25  * \class scGenComp_PU_Tech
26  * \brief A simple abstract class to implement the operation of a general Tech computing unit
27  *
28  * Operating principle of event processing:
29  *
30  * The non-virtual "XXX_method"s are "sensitive" to specific events. After receiving such an an event,
31  * the corresponding virtual routine of "XXX_Do" is called.
32  * If the unit's state and the call are not compatible (programming mistake), that routine asserts.
33  * (wrong event)
34  * The time delay of the operation is simulated by issuing a wait() call to SystemC kernel.
35  * Optionally, also the next state is invoked by issuing a EVENT_GenComp.XXX.notify() notification.
36  * The description is valid for methods
37  * - Deliver
38  * - Process
39  * - Relax
40  * - Synchronize
41  *
42  * @see AbstractGenCompState
43  *
44  * The rest of methods are of technical nature.
45  * - HeartBeat: technical signal to update PU's state (e.g. integrate a signal)
46  * - Fail: something went wrong, retry
47  * - Sleep: if the unit is unused, sends it logically to sleep
48  * - Wakeup: awake it if it was sleeping
49  *
50  * Operating mode:
51  * - in synchronized mode, an external clock signal advances states
52  * from "Relaxing' to 'Computing', and 'Computing' to 'Delivering', then autonomously to 'Relaxing'
53  * - in asynchron mode, an internal, per module, 'Heartbeat' signal is used to advance from state to state
54  * - in asynchron mode , the operations can be of fixed time
55  * (such as fetch time or execution time in technical systems) or variable time (such as periodic checking
56  * of variable length of states depending on analog quantities in biological systems)
57  * when heartbeat serves for periodic checking of states),
58  * here heartbeat routines provide state change transition signals
59  *
60  * 'Computing' and 'Delivering' can be done in parallel ('Relative refractory' in biology,
61  * pre-fetching in technology)
62  */
